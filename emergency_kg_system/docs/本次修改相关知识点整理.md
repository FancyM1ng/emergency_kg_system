# 本次修改相关知识点整理

## 1. Neo4j 中“来源文件显示未知”的根因

### 现象
- Web 页面中“查看知识来源”时，来源文件显示为“未知来源”。

### 根因
- 三元组导入时，关系和节点都写了 `source`。
- 但问答检索最初优先读取的是节点上的 `source`，而不是关系上的 `source`。
- 节点会在多次导入中被重复 `MERGE` 和 `SET`，所以节点上的 `source` 很容易被后续文档覆盖，不适合作为精确来源。
- 真正稳定的来源应该放在关系上，因为一条关系对应一条具体三元组。

### 正确做法
- 导入时把来源写到关系属性 `r.source`。
- 查询时优先读取 `r.source`，节点来源只作为兜底。

### 本项目中的实现原则
- 导入脚本：`r.source = $source`
- 问答脚本：`coalesce(r.source, h.source, t.source) as source`

### 经验结论
- 知识图谱中如果需要追踪“三元组来自哪篇文档”，优先把来源挂在关系上，而不是节点上。

## 2. Neo4j 关系类型不能直接使用任意中文字符串

### 现象
- 导入时报错，关系名里包含 `、`、空格、斜杠等字符时，Cypher 语句会出现语法错误。

### 原因
- Neo4j 的关系类型属于 Cypher 标识符，不能直接包含任意特殊字符。

### 解决方式
- 导入前对关系名做清洗。
- 将非法字符替换为下划线。
- 如果清洗后为空，则使用默认关系名，例如 `RELATED_TO`。
- 原始关系文本单独保存到 `r.original_relation`，用于展示和问答。

### 经验结论
- `type(r)` 用于数据库内部关系类型。
- `r.original_relation` 用于界面展示和保留原始语义。

## 3. 三元组抽取结果应按文件分别保存

### 之前的问题
- 所有文本文件切片后，结果统一写入一个总 JSON。
- 后续导入和溯源时，不方便定位某条三元组属于哪个源文件。

### 调整后的思路
- 每次只处理一个 `txt` 文件。
- 处理完成后，将该文件对应的所有三元组保存为“同名 JSON 文件”。
- 例如：
  - `国家级预案_xxx.txt`
  - `国家级预案_xxx.json`

### 这样做的好处
- 文件级管理更清晰。
- 导入 Neo4j 时更容易按文件批量处理。
- 后续排查 LLM 提取问题、重复数据问题时更方便。

## 4. 历史记录不应放在 annotations 目录

### 原因
- `data/annotations/` 更适合存放标注数据、抽取结果、三元组文件。
- 聊天历史属于运行时数据，不应该和标注结果混在一起。

### 更合理的目录设计
- 标注/抽取结果：`data/annotations/`
- 聊天历史：`data/history/`

### 本次调整
- 将历史记录文件迁移到：
  - `data/history/chat_history.json`
- 保留对旧路径 `data/annotations/chat_history.json` 的兼容读取和迁移逻辑。

### 经验结论
- 目录分层要按“数据职责”划分，而不是按“都能存 JSON”来划分。

## 5. 历史记录中的旧数据为什么仍然没有来源

### 原因
- 旧历史记录是在来源字段还未正确保存时生成的。
- 那些记录中的 `knowledge` 项里没有 `source`，因此页面显示时会是空。

### 解决思路
- 新问答结果直接写入 `source`。
- 对旧历史记录做“懒回填”：
  - 页面读取历史时，根据 `head + relation + tail` 回查 Neo4j
  - 若找到对应关系，则把 `source` 补回历史记录

### 经验结论
- 历史数据结构升级后，最好做兼容迁移或懒加载修复，否则旧数据会长期不完整。

## 6. BERT 模型应做本地持久化缓存

### 之前的问题
- `bert-base-chinese` 初次加载会从 Hugging Face 下载。
- 如果没有指定缓存目录，后续环境变化或缓存位置不稳定时，容易重复下载。

### 调整后的做法
- 显式指定本地缓存目录，例如：
  - `data/model_cache/huggingface`
- 初始化模型时通过 `cache_dir` 传入。

### 典型实现思路
```python
self.tokenizer = BertTokenizer.from_pretrained(
    model_name,
    cache_dir=str(self.cache_dir),
)
self.model = BertModel.from_pretrained(
    model_name,
    cache_dir=str(self.cache_dir),
)
```

### 好处
- 避免重复下载。
- 项目迁移时缓存位置更可控。
- 更适合写入自己的工程知识库和部署文档。

## 7. Streamlit 页面优化的几个关键点

### 7.1 大模型和 BERT 要延迟初始化

#### 原因
- 如果页面一启动就初始化问答系统和 BERT，会导致首页打开很慢。

#### 思路
- 只有用户真正点击问答功能时，才初始化问答系统。

#### 经验结论
- 对于 Streamlit 项目，重资源对象应延迟加载或缓存。

### 7.2 图谱页应补充图例

#### 原因
- 非技术用户看到不同颜色节点时，不知道颜色代表什么。

#### 调整内容
- 在图谱可视化底部增加颜色说明区。
- 说明红色、绿色、蓝色、紫红、金色节点分别代表什么。
- 同时说明箭头和边标签的含义。

### 7.3 图谱显示边数量不应写死

#### 之前的问题
- 滑块范围固定为 `30-300`，当数据库中实际关系数大于 300 时不合理。

#### 调整方式
- 滑块范围改为：
  - 最小值：30
  - 最大值：当前数据库中的总关系数

#### 经验结论
- 可视化参数上限最好绑定真实数据规模，而不是写死常量。

## 8. AuraDB 中“显示导入成功但看不到数据”的排查思路

### 核心检查项
- `.env` 中的 `NEO4J_URI`
- `.env` 中的 `NEO4J_USER`
- `.env` 中的 `NEO4J_PASSWORD`
- `.env` 中的 `NEO4J_DATABASE`

### 典型问题
- 导入脚本和查询脚本使用的数据库名不一致。
- 数据导入到了一个数据库，但页面查询的是另一个数据库。

### 经验结论
- 在 AuraDB 中，数据库名必须统一。
- 导入、查询、可视化三个模块必须共用同一份连接配置。

## 9. Git 提交与推送的本质区别

### 提交 commit
- 只把改动保存到本地 Git 仓库。
- 这一步不会上传到 GitHub。

### 推送 push
- 把本地分支的提交上传到远程仓库 GitHub。

### “发布 branch” 的含义
- 当前本地分支还没有推送到远程。
- 点击“发布 branch”本质上就是第一次执行推送并建立远程跟踪关系。

### 判断是否已经上传成功
```powershell
git branch -vv
git status -sb
```

### 成功推送后的典型结果
```powershell
* main 5fd3869 [origin/main] 完善项目
## main...origin/main
```

## 10. 为什么 `git push` 之前失败，后来又成功

### 之前的报错
```powershell
fatal: unable to access 'https://github.com/FancyM1ng/emergency_kg_system.git/':
Recv failure: Connection was reset
```

### 这类报错说明什么
- 不是仓库地址错误。
- 不是不会操作。
- 而是 Git 通过 HTTPS 与 GitHub 通信时，连接在认证前后被重置。

### 已确认过的事实
- 电脑可以访问 `https://github.com`
- `github.com:443` 端口是通的
- 远程地址配置正确

### 真正的问题
- Git for Windows 默认的 HTTPS/TLS 通信方式，在当前环境下兼容性不佳。

### 为什么修改后就成功了
执行了：
```powershell
git config --global http.version HTTP/1.1
git config --global http.sslBackend schannel
```

### 两条配置的含义
- `http.version HTTP/1.1`
  - 让 Git 使用更稳定的 HTTP/1.1 通信方式
- `http.sslBackend schannel`
  - 让 Git 使用 Windows 自带的证书和 TLS 通道，而不是 Git 自己的 SSL 实现

### 结果
- Git 与 GitHub 的连接更稳定
- 认证流程可以正常启动
- 所以才弹出 GitHub 登录页面
- 登录成功后推送也成功

### 经验结论
- 如果浏览器能打开 GitHub，但 `git push` 报 `Connection was reset`，优先考虑 Git 的 HTTPS/TLS 配置兼容性问题。

## 11. Windows 控制台编码问题

### 现象
- Python 脚本中打印 emoji 时，PowerShell 可能报：
```text
UnicodeEncodeError: 'gbk' codec can't encode character
```

### 原因
- Windows 控制台默认可能仍然使用 GBK 编码。
- emoji 或某些 Unicode 字符无法正常输出。

### 经验结论
- 调试脚本中的 `print` 尽量避免使用 emoji。
- 如果必须使用，可考虑切换终端编码为 UTF-8。

## 12. 本次可复用的工程经验总结

### 数据建模层面
- 溯源信息优先挂在关系上，不挂在节点上。
- 展示字段和数据库内部字段应分离，例如：
  - 内部关系类型：`type(r)`
  - 展示关系文本：`r.original_relation`

### 数据组织层面
- 原始文本、抽取结果、运行历史应分目录管理。
- 单文件单结果，便于追踪和重导。

### 页面与交互层面
- 延迟加载重资源。
- 图例、来源、历史等辅助信息要面向非技术用户设计。
- 界面参数上限应使用真实数据动态计算。

### 运维与排障层面
- 先区分“本地提交”和“远程推送”。
- 网络问题要区分：
  - 浏览器能不能打开
  - 端口通不通
  - Git 自己的 HTTPS/TLS 配置是否兼容
